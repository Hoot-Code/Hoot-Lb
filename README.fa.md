# Hoot-Lb

یک بالانسر بار (load balancer) سبک و بدون وابستگی خارجی، نوشته‌شده با
Go — جایگزینی سبک‌تر برای NGINX/HAProxy/Traefik، با پشتیبانی از
بالانس بار در لایه ۴ (TCP/UDP) و لایه ۷ (HTTP).

## مقایسه با بالانسرهای بار دیگر

| ویژگی | Hoot-Lb | NGINX | HAProxy | Traefik |
|--------|---------|-------|---------|---------|
| اندازه باینری | ~۸ مگابایت | ~۲ مگابایت | ~۲ مگابایت | ~۳۰ مگابایت |
| وابستگی‌های خارجی | ۰ (فقط stdlib) | بسیار | بسیار | بسیار |
| فرمت تنظیمات | YAML (۱ فایل) | nginx.conf | haproxy.cfg | YAML/TOML/labels |
| L4 TCP/UDP | بله | بله | بله | فقط TCP |
| L7 HTTP | بله | بله | بله | بله |
| بارگذاری مجدد داغ | بله (SIGHUP/polling) | بله (signal) | بله (socket) | بله (dynamic) |
| ریستارت بدون داون‌تایم | بله (FD handoff) | نه | بله (runtime API) | بله |
| کشف سرویس | DNS, Consul | DNS, Consul (commercial) | DNS, Consul, etcd | Docker, K8s, Consul, etcd |
| داشبورد | داخلی | شخص ثالث | Stats socket | داخلی |
| محدودیت نرخ | Token bucket به‌ازای هر کلاینت | Limit_req zone | Stick-table | InFlightReq |
| قطع مدار | داخلی | نه | نه | بله |
| Admin API | REST + WebSocket | nginx plus ($) | Runtime API | API |

Hoot-Lb به‌جای توان خام، سادگی را انتخاب می‌کند: صفر فایل تنظیمات،
صفر مدیر بسته، صفر وابستگی‌های زمان بیلد. کل باینری فقط از
کتابخانه استاندارد Go ساخته می‌شود.

## ساخت و اجرا

این پروژه **هیچ وابستگی خارجی‌ای ندارد** — فقط کتابخانه استاندارد،
به‌صورت عمدی. نیاز به Go نسخه ۱.۲۲ یا بالاتر دارد.

```sh
go build ./...
go vet ./...

go run ./cmd/lb -config examples/config.yaml
```

برای خاموش‌کردن، سیگنال SIGINT/SIGTERM (مثلاً Ctrl-C) ارسال کنید.

## تنظیمات (Configuration)

برای یک نمونه کامل با توضیحات به فایل `examples/config.yaml` مراجعه
کنید. این طرح سه بخش اصلی دارد:

- `global` — تنظیمات سراسری پردازه (سطح لاگ، زمان خاموش‌شدن
  آبرومندانه).
- `listeners` — لیسنرهای شبکه‌ای که باز می‌شوند (نام، آدرس، پروتکل،
  اینکه به کدام pool مسیر می‌دهند، و تنظیمات اختیاری TLS).
- `pools` — گروه‌های نام‌گذاری‌شده‌ای از بک‌اندهای وزن‌دار که
  لیسنرها ترافیک را به آن‌ها مسیر می‌دهند. هر pool یک الگوریتم
  بالانس بار را از طریق فیلد `algorithm` انتخاب می‌کند و به‌صورت
  اختیاری سلامت‌سنجی را با فیلد `health_check` و نشست‌های پایدار را
  با فیلد `sticky` پیکربندی می‌کند.

### مسیریابی HTTP

لیسنرهای HTTP از مسیریابی اختیاری بر اساس host/path/header از طریق
فیلد `routes` پشتیبانی می‌کنند. وقتی این فیلد موجود باشد، درخواست‌های
ورودی پیش از بازگشت به `pool` پیش‌فرض لیسنر، با این مسیرها تطبیق
داده می‌شوند.

```yaml
listeners:
  - name: web
    address: "0.0.0.0:8080"
    protocol: http
    pool: web_pool
    routes:
      - host: api.example.com
        path_prefix: /v2
        pool: api_pool
      - path_prefix: /static
        pool: static_pool
      - path_prefix: /beta
        header:
          name: X-Beta-User
          value: "true"
        pool: beta_pool
```

مسیرها بر اساس طولانی‌ترین `path_prefix` در میان مسیرهایی که شرط‌های
`host` و `header` آن‌ها تطبیق دارد انتخاب می‌شوند (host خالی با هر
host درخواست تطبیق دارد، و عدم وجود شرط header با هر هدری تطبیق
دارد). اگر چند مسیر طول prefix یکسانی داشته باشند، اولین مسیر در
ترتیب config برنده می‌شود. اگر هیچ مسیری تطبیق نداشت، `pool`
پیش‌فرض لیسنر استفاده می‌شود.

همه شرط‌های یک مسیر با AND ترکیب می‌شوند: host، path_prefix، و
header باید همگی تطبیق داشته باشند تا آن مسیر تطبیق پیدا کند.

فیلد `routes` فقط برای لیسنرهای HTTP معتبر است — مشخص‌کردن آن برای
لیسنرهای TCP یا UDP یک خطای اعتبارسنجی است.

#### تطبیق هدر

مسیرها می‌توانند یک شرط اختیاری `header` برای تطبیق دقیق نام+مقدار
با هدرهای درخواست ورودی داشته باشند. نام هدر بدون توجه به بزرگی/کوچکی
حروف تطبیق داده می‌شود (طبق semantics HTTP، که `net/http.Header.Get`
به‌صورت پیش‌فرض همین کار را می‌کند).

```yaml
routes:
  - path_prefix: /beta
    header:
      name: X-Beta-User
      value: "true"
    pool: beta_pool
```

### تقسیم ترافیک کاناری (Canary)

به‌جای مسیردهی به یک pool واحد، یک مسیر می‌تواند یک `split` تعریف
کند — فهرستی وزن‌دار از pool‌ها برای توزیع ترافیک به‌سبک کاناری.
ترافیک متناسب با وزن، از طریق انتخاب تصادفی به‌ازای هر درخواست توزیع
می‌شود. فیلدهای `pool` و `split` روی یک مسیر متقابلاً منحصرند.

```yaml
routes:
  - path_prefix: /checkout
    split:
      - pool: checkout_stable
        weight: 90
      - pool: checkout_canary
        weight: 10
```

**قوانین اعتبارسنجی برای split:**

- باید حداقل ۲ ورودی داشته باشد.
- همه وزن‌ها باید عدد صحیح مثبت باشند.
- هر pool ارجاع‌شده باید در بخش `pools` وجود داشته باشد.
- `pool` و `split` نمی‌توانند هر دو روی یک مسیر مشخص شوند.

### نشست‌های پایدار (Sticky sessions)

Pool‌ها می‌توانند نشست‌های پایدار مبتنی بر کوکی را از طریق بلوک
`sticky` فعال کنند. وقتی فعال باشد، پراکسی کوکی‌ای تنظیم می‌کند که
بک‌اند انتخاب‌شده برای یک درخواست را مشخص می‌کند. درخواست‌های بعدی
که آن کوکی را حمل می‌کنند مستقیماً به همان بک‌اند (اگر هنوز در pool
و سالم باشد) مسیر می‌شوند، و الگوریتم بالانس بار pool را نادیده
می‌گیرند.

```yaml
pools:
  - name: web_pool
    sticky:
      cookie_name: hoot_lb_sticky    # پیش‌فرض "hoot_lb_sticky"
      ttl: 1h                        # Max-Age کوکی؛ پیش‌فرض 1h
    backends:
      - address: "10.0.0.1:8080"
        weight: 1
      - address: "10.0.0.2:8080"
        weight: 1
```

**رفتار:**

- **کوکی sticky موجود و معتبر**: مستقیماً به بک‌اند قبلاً انتخاب‌شده
  مسیر می‌دهد (اگر هنوز در pool و سالم باشد).
- **بدون کوکی / کوکی نامعتبر / بک‌اند حذف‌شده یا ناسالم**: به `Pick`
  عادی برمی‌گردد و یک کوکی sticky تازه تنظیم می‌کند.
- **sticky + ip_hash**: وقتی کوکی معتبری موجود باشد، sticky نسبت به
  ip_hash اولویت دارد. در یک بازدید تازه، Pick به‌صورت عادی اجرا
  می‌شود (که ممکن است خود ip_hash باشد)، و نتیجه آن بک‌اند تازه
  "چسبیده" می‌شود.

ویژگی‌های کوکی: `Secure`، `HttpOnly`، `SameSite=Lax`، `Path=/`،
`Max-Age` مطابق با TTL پیکربندی‌شده. مقدار کوکی همان آدرس بک‌اند
است — هیچ امضای رمزنگاری‌شده‌ای لازم نیست چون مقدار کوکی فقط پس از
تأیید عضویت فعلی بک‌اند در pool قابل اعتماد است.

**اعتبارسنجی:** وقتی `sticky` موجود باشد، `ttl` باید مثبت باشد.

### TLS

لیسنرها می‌توانند به‌صورت اختیاری TLS را از طریق بلوک `tls` فعال
کنند. دو حالت پشتیبانی می‌شود:

#### پایان‌دهی TLS (لیسنرهای HTTP)

پراکسی ترافیک TLS ورودی را رمزگشایی می‌کند و HTTP ساده را به
بک‌اندها فوروارد می‌کند. کلاینت گواهی پراکسی را می‌بیند؛ بک‌اند
هرگز ترافیک رمزشده را نمی‌بیند.

```yaml
listeners:
  - name: web_tls
    address: "0.0.0.0:8443"
    protocol: http
    pool: web_pool
    tls:
      mode: terminate
      certificates:
        - host: "example.com"       # تطبیق SNI
          cert_file: "/path/cert.pem"
          key_file: "/path/key.pem"
        - host: ""                   # خالی = گواهی پیش‌فرض/fallback
          cert_file: "/path/default-cert.pem"
          key_file: "/path/default-key.pem"
```

- **`mode: terminate`** — نیازمند `protocol: http` است.
- **`certificates`** — حداقل یک ورودی لازم است. هر ورودی یک نام
  میزبان SNI را به یک زوج فایل cert/key نگاشت می‌کند. حداکثر یک
  ورودی می‌تواند `host` خالی داشته باشد، که به‌عنوان پیش‌فرض هنگام
  نبود یا عدم تطبیق SNI استفاده می‌شود.

#### عبور SNI تی‌ال‌اس (لیسنرهای TCP)

پراکسی پیام ClientHello تی‌ال‌اس را برای استخراج نام میزبان SNI
بررسی می‌کند، اتصال را به pool بک‌اند مناسب مسیر می‌دهد، و بایت‌های
اصلی رمزشده را بدون تغییر رله می‌کند. پراکسی هرگز کلید خصوصی‌ای
نگه نمی‌دارد و هرگز ترافیک را رمزگشایی نمی‌کند — بک‌اند handshake
TLS خودش را مستقیماً با کلاینت انجام می‌دهد.

```yaml
listeners:
  - name: passthrough_tls
    address: "0.0.0.0:9443"
    protocol: tcp
    pool: default_pool              # fallback وقتی SNI تطبیق ندارد
    tls:
      mode: passthrough
      routes:
        - host: "a.example.com"
          pool: pool_a
        - host: "b.example.com"
          pool: pool_b
```

- **`mode: passthrough`** — نیازمند `protocol: tcp` است.
- **`routes`** — نام‌های میزبان SNI را به pool‌های بک‌اند نگاشت
  می‌کند. فیلد `pool` خود لیسنر به‌عنوان fallback وقتی هیچ مسیری
  تطبیق ندارد یا SNI موجود نیست استفاده می‌شود.

### الگوریتم‌های بالانس بار

فیلد `algorithm` روی هر pool استراتژی بالانس بار را انتخاب می‌کند.
باید یکی از موارد زیر باشد:

- **`round_robin`** — round-robin ساده و بدون وزن. به‌ترتیب بین
  بک‌اندها می‌چرخد و وزن را نادیده می‌گیرد. مناسب وقتی همه بک‌اندها
  برابرند.
- **`weighted_round_robin`** (پیش‌فرض) — round-robin وزن‌دار هموار
  با استفاده از الگوریتم interleaved. بک‌اندهای با وزن بالاتر،
  ترافیک بیشتری به‌نسبت دریافت می‌کنند بدون انباشته‌شدن ناهموار.
- **`least_conn`** — بک‌اندی با کمترین تعداد اتصال فعال (in-flight)
  را انتخاب می‌کند. از وزن به‌عنوان تساوی‌شکن استفاده می‌کند. نیازمند
  این است که لایه پراکسی Acquire/Release را از طریق رابط ConnReleaser
  صدا بزند.
- **`random`** — انتخاب تصادفی یکنواخت با احتمال متناسب با وزن.
  مناسب وقتی یک الگوریتم ساده و بدون حالت (stateless) کافی است.
- **`ip_hash`** — انتخاب قطعی مبتنی بر هش. یک IP کلاینت مشخص همیشه
  به همان بک‌اند نگاشت می‌شود (تا زمانی که مجموعه بک‌اندها تغییر
  نکند). برای چسبندگی نشست (session affinity) هم در لایه ۴ و هم
  لایه ۷ مفید است.

وقتی حذف شود، `algorithm` به‌صورت پیش‌فرض `weighted_round_robin`
خواهد بود.

### سلامت‌سنجی (Health checking)

هر pool می‌تواند به‌صورت اختیاری سلامت‌سنجی را از طریق بلوک
`health_check` پیکربندی کند. وقتی حذف شود، pool به‌صورت پیش‌فرض از
سلامت‌سنجی TCP با آستانه‌های پیش‌فرض استفاده می‌کند.

```yaml
health_check:
  type: tcp              # tcp | http | none (پیش‌فرض: tcp)
  path: /healthz         # فقط وقتی type: http لازم است
  interval: 5s           # پیش‌فرض 5s
  timeout: 2s            # پیش‌فرض 2s
  healthy_threshold: 2   # تعداد موفقیت‌های پیاپی برای علامت‌گذاری سالم (پیش‌فرض 2)
  unhealthy_threshold: 3 # تعداد شکست‌های پیاپی برای علامت‌گذاری ناسالم (پیش‌فرض 3)
```

**فیلدها:**

- **`type`** — `tcp` با باز کردن یک اتصال TCP بررسی می‌کند (بررسی
  زنده‌بودن). `http` با ارسال یک درخواست GET به `path` بررسی می‌کند
  و انتظار پاسخ 2xx دارد. `none` همه سلامت‌سنجی را برای pool غیرفعال
  می‌کند — بک‌اندها همیشه سالم باقی می‌مانند. این یک راه فرار عمدی
  برای بک‌اندهایی است که از بررسی‌های استاندارد پشتیبانی نمی‌کنند.
- **`path`** — مسیر HTTP برای بررسی وقتی `type` برابر `http` است.
  باید با `/` شروع شود. وقتی `type` برابر `http` است الزامی است.
- **`interval`** — هر بک‌اند هر چند وقت بررسی می‌شود. باید مثبت باشد.
- **`timeout`** — حداکثر زمان انتظار برای یک بررسی. باید مثبت و
  کمتر از `interval` باشد.
- **`healthy_threshold`** — تعداد بررسی‌های موفق پیاپی لازم برای
  علامت‌گذاری یک بک‌اند ناسالم به‌عنوان سالم. باید >= 1 باشد.
- **`unhealthy_threshold`** — تعداد بررسی‌های ناموفق پیاپی لازم برای
  علامت‌گذاری یک بک‌اند سالم به‌عنوان ناسالم. باید >= 1 باشد.

**بررسی فعال در مقابل غیرفعال:**

- بررسی **فعال** یک goroutine به‌ازای هر بک‌اند اجرا می‌کند، که هر
  کدام با فاصله‌زمانی پیکربندی‌شده و با تأخیرهای اولیه پلکانی (برای
  جلوگیری از انبوه بررسی‌های هم‌زمان) بررسی می‌کنند. از آستانه‌های
  hysteresis برای جلوگیری از flapping استفاده می‌شود — یک شکست
  منفرد هرگز فوراً یک بک‌اند سالم را ناسالم علامت نمی‌زند، و برعکس.
- بررسی **غیرفعال (passive)** فوراً زیرسیستم سلامت را زمانی که یک
  dial واقعی از طرف کلاینت شکست می‌خورد (نه یک probe مصنوعی) آگاه
  می‌کند. یک شکست dial منفرد فوراً بک‌اند را ناسالم علامت می‌زند —
  بدون آستانه. این یک شکست تأییدشده روی ترافیک واقعی است. بازیابی
  همچنان نیازمند `healthy_threshold` موفقیت فعال پیاپی است.

**قوانین اعتبارسنجی:**

- `interval` و `timeout` باید مثبت باشند.
- `timeout` باید کمتر از `interval` باشد.
- هر دو آستانه باید >= 1 باشند.
- وقتی `type` برابر `http` است، `path` الزامی است و باید با `/`
  شروع شود.
- `type` باید یکی از این موارد باشد: `tcp`، `http`، `none`.

اعتبارسنجی در زمان بارگذاری انجام می‌شود؛ خطاها فیلد دقیق اشتباه را
نام می‌برند (مثلاً `pools[0].health_check.timeout: must be less than
interval`) به‌جای شکست عمومی.

نکته: به‌صورت عمدی هیچ وابستگی به کتابخانه YAML خارجی وجود ندارد، پس
`internal/config` یک پارسر کوچک برای زیرمجموعه‌ای از YAML که این طرح
نیاز دارد (نگاشت‌های تودرتو، دنباله‌هایی از نگاشت‌ها/مقادیر ساده،
رشته‌های با علامت نقل‌قول، کامنت‌ها) شامل می‌شود. این یک پارسر
عمومی YAML نیست — پیش از گسترش طرح با ویژگی‌های YAML که هنوز پشتیبانی
نمی‌شوند، کامنت بالای `internal/config/yaml.go` را بخوانید.

## بارگذاری مجدد داغ تنظیمات (Hot reload)

بالانسر بار از بارگذاری مجدد داغ بدون قطعی برای برخی تغییرات
پیکربندی، بدون نیاز به راه‌اندازی مجدد پردازه، پشتیبانی می‌کند. این
بازخوانی توسط موارد زیر فعال می‌شود:

- **Polling فایل**: فایل config با یک فاصله‌زمانی قابل‌تنظیم (پیش‌فرض
  ۵ ثانیه) بررسی می‌شود. وقتی mtime و اندازه فایل تغییر کند،
  پیکربندی جدید بارگذاری و اعتبارسنجی می‌شود.
- **سیگنال SIGHUP**: با `kill -HUP <pid>` بدون انتظار برای tick بعدی
  polling، بازخوانی فوری انجام می‌شود.

### چه چیزی قابل بارگذاری مجدد داغ است

- **ترکیب pool**: آدرس‌های بک‌اند، وزن‌ها، الگوریتم بالانس بار،
  پیکربندی سلامت‌سنجی، و تنظیمات نشست پایدار.
- **مسیرها و splitها**: جدول‌های مسیر لیسنر HTTP (تطبیق host، path،
  header، splitهای کاناری) از روی config جدید بازسازی می‌شوند.
- **گواهی‌های TLS**: فایل‌های گواهی برای لیسنرهای TLS-terminate از
  دیسک از طریق مکانیزم CertStore اتمیک موجود بازخوانی می‌شوند.

### چه چیزی نیازمند راه‌اندازی مجدد است

- **تغییرات سطح لیسنر**: افزودن/حذف یک لیسنر، یا تغییر آدرس،
  پروتکل، یا حالت TLS آن نیازمند راه‌اندازی مجدد کامل پردازه است.
  این به‌صراحت در طول بازخوانی با یک پیام خطای واضح رد می‌شود.
  از SIGUSR2 یا `POST /admin/restart` برای ریستارت بدون قطعی
  استفاده کنید.

### تنظیمات

```yaml
global:
  log_level: info
  shutdown_timeout: 10s
  reload_check_interval: 5s  # صفر یا حذف‌شده polling را غیرفعال می‌کند
  max_connections_per_listener: 1000  # 0 = بدون محدودیت
  tcp_idle_timeout: 30s               # 0 = غیرفعال
```

- `reload_check_interval`: هر چند وقت فایل config برای تغییرات
  بررسی می‌شود. هر رشته مدت‌زمانی را می‌پذیرد (مثلاً `"5s"`،
  `"30s"`). وقتی صفر یا حذف‌شده باشد، polling فایل غیرفعال می‌شود و
  فقط SIGHUP باعث بازخوانی می‌شود.
- `max_connections_per_listener`: حداکثر تعداد اتصالات همزمان در هر
  لیسنر TCP یا HTTP را محدود می‌کند. وقتی لیسنر به این حد رسید،
  اتصالات جدید فوراً بسته می‌شوند. لیسنرهای UDP حذف شده‌اند.
- `tcp_idle_timeout`: حداکثر زمان بیکاری یک اتصال TCP یا TLS
  passthrough قبل از بسته شدن. ددلاین هر بار که داده‌ای ارسال شود
  ریست می‌شود.

### رفتار بازخوانی

۱. پیکربندی جدید خوانده و پارس می‌شود.
۲. اعتبارسنجی کامل اجرا می‌شود (همان pipeline زمان راه‌اندازی — هیچ
   مسیر اعتبارسنجی سبک‌تری برای بازخوانی وجود ندارد).
۳. مجموعه لیسنرها با پیکربندی در حال اجرا دیف (diff) می‌شود. هر
   تغییری در سطح لیسنر کل بازخوانی را رد می‌کند (همه یا هیچ).
۴. برای لیسنرهای TLS-terminate، گواهی‌های جدید بارگذاری و به‌صورت
   اتمیک در CertStore موجود جابه‌جا می‌شوند.
۵. یک Snapshot جدید با LoadBalancerها، Serverها، جدول‌های مسیر، و
   HealthCheckerهای تازه ساخته می‌شود.
۶. HealthCheckerهای جدید شروع می‌شوند.
۷. اشاره‌گر Snapshot به‌صورت اتمیک جابه‌جا می‌شود — همه اتصالات/
   درخواست‌های جدید فوراً وضعیت جدید را می‌بینند.
۸. HealthCheckerهای قدیمی متوقف می‌شوند (پس از جابه‌جایی، هرگز پیش
   از آن).

در صورت بروز خطا در مراحل ۱ تا ۴، خطا لاگ می‌شود و پیکربندی فعلی
بدون تغییر باقی می‌ماند.

## کشف سرویس (Service discovery)

Pool‌ها می‌توانند به‌جای یک فهرست بک‌اند ثابت، از کشف سرویس استفاده
کنند. بلوک `discovery` روی یک pool یک منبع بک‌اند پویا را پیکربندی
می‌کند که با یک فاصله‌زمانی پیکربندی‌شده poll می‌شود.

### کشف مبتنی بر DNS

```yaml
pools:
  - name: web_pool
    algorithm: weighted_round_robin
    discovery:
      type: dns
      dns:
        host: "backend.service.consul"
        port: 8080
        refresh_interval: 30s
    health_check:
      type: tcp
```

**نکته TTL در DNS:** resolver کتابخانه استاندارد Go مقادیر واقعی TTL
رکورد DNS را از طریق API `LookupHost` در معرض قرار نمی‌دهد. اپراتور
**باید** `refresh_interval` را برابر یا کمتر از TTL واقعی DNS خود
تنظیم کند. این سیستم نمی‌تواند فاصله‌زمانی refresh درست را به‌صورت
خودکار تشخیص دهد.

**رفتار وزن:** کشف DNS وزن‌های به‌ازای هر بک‌اند را کدگذاری نمی‌کند.
هر بک‌اند کشف‌شده به‌صورت پیش‌فرض وزن ۱ دارد. وقتی الگوریتم pool
`weighted_round_robin` باشد، وزن‌ها در عمل بین همه بک‌اندهای
کشف‌شده یکسان است — این رفتار مورد انتظار است، نه یک باگ.

### کشف مبتنی بر Consul

کشف Consul فقط زمانی موجود است که با تگ بیلد `consul` ساخته شود:

```sh
go build -tags consul ./...
```

```yaml
pools:
  - name: web_pool
    algorithm: weighted_round_robin
    discovery:
      type: consul
      consul:
        address: "http://127.0.0.1:8500"
        service: "web"
        refresh_interval: 30s
    health_check:
      type: tcp
```

اگر config مقدار `type: consul` را مشخص کند اما باینری بدون
`-tags consul` ساخته شده باشد، اعتبارسنجی config با یک خطای واضح
شکست می‌خورد.

### رفتار کشف سرویس

- یک pool یا بک‌اندهای ثابت `backends` دارد یا یک بلوک `discovery`
  — هرگز هر دو با هم (خطای اعتبارسنجی config).
- خطاهای کشف هرگز فهرست بک‌اند یک pool را خالی نمی‌کنند. pool به
  سرو کردن آخرین موردی که با موفقیت resolve شده ادامه می‌دهد.
- با تغییر مجموعه بک‌اند، فقط pool تأثیرپذیرفته بازسازی می‌شود.
  pool‌های دیگر دست‌نخورده باقی می‌مانند (به‌روزرسانی جزئی از طریق
  identity اشاره‌گر).
- HealthCheckerها در طول به‌روزرسانی‌های جزئی از ترتیب
  شروع-قبل-از-توقف پیروی می‌کنند.

## REST API مدیریتی (Admin API)

Hoot-Lb یک REST API مدیریتی برای عملیات صفحه کنترل در زمان اجرا
فراهم می‌کند. به‌صورت پیش‌فرض فقط روی loopback بایند می‌شود
(`127.0.0.1:9091`).

یک داشبورد وب زنده روی همین لیسنر سوار می‌شود — به بخش
[داشبورد](#داشبورد) در پایین مراجعه کنید. هیچ بلوک تنظیمات جدایی
برای داشبورد وجود ندارد: فعال‌کردن سرور ادمین داشبورد را هم فعال
می‌کند، و غیرفعال‌کردن آن هر دو را غیرفعال می‌کند.

### تنظیمات

```yaml
global:
  admin:
    enabled: true
    address: "127.0.0.1:9091"
    token_env: "HOOT_LB_ADMIN_TOKEN"
    max_concurrent_requests: 10
```

**مهم:** توکن ادمین از متغیر محیطی‌ای که در `token_env` نام برده
شده خوانده می‌شود. هرگز توکن را به‌صورت متن صریح در فایل config قرار
ندهید. متغیر محیطی نام‌برده‌شده باید در زمان راه‌اندازی موجود و
غیرخالی باشد — در غیر این صورت پردازه با یک خطای واضح فوراً شکست
می‌خورد.

### احراز هویت

همه نقاط پایانی ادمین نیازمند یک توکن bearer از طریق هدر
`Authorization` هستند:

```
Authorization: Bearer <token>
```

توکن با مقایسه با زمان ثابت (constant-time) مقایسه می‌شود تا از
حملات کانال‌جانبی زمان‌بندی (timing side-channel) جلوگیری شود. این
برای همه نقاط پایانی، حتی نقاط فقط-خواندنی، به‌منظور یک مدل ذهنی
یکدست اعمال می‌شود.

### نقاط پایانی

| متد | مسیر | بدنه | توضیح |
|------|------|------|-------|
| GET | `/admin/pools` | — | فهرست همه pool‌ها با وضعیت بک‌اند |
| POST | `/admin/pools/{pool}/backends` | `{"address":"...","weight":N}` | افزودن یک بک‌اند |
| DELETE | `/admin/pools/{pool}/backends/{address}` | — | حذف یک بک‌اند |
| POST | `/admin/pools/{pool}/backends/{address}/drain` | — | تنظیم وضعیت drain |
| POST | `/admin/pools/{pool}/backends/{address}/undrain` | — | پاک‌کردن وضعیت drain |

**پاسخ GET /admin/pools:**

```json
{
  "pools": [{
    "name": "web_pool",
    "algorithm": "weighted_round_robin",
    "discovery": false,
    "backends": [{
      "address": "10.0.0.1:8080",
      "weight": 5,
      "healthy": true,
      "draining": false
    }]
  }]
}
```

**پاسخ‌های خطا:**

| وضعیت | معنا |
|--------|------|
| 401 | توکن bearer گم‌شده یا نامعتبر |
| 404 | pool یا بک‌اند پیدا نشد |
| 409 | بک‌اند از قبل وجود دارد، یا pool از discovery استفاده می‌کند (افزودن/حذف) |
| 503 | از حد هم‌زمانی عبور شده |

### رد درخواست برای pool‌های مبتنی بر discovery

افزودن/حذف بک‌اند توسط ادمین فقط برای pool‌هایی با پیکربندی ثابت
`backends` معتبر است. pool‌هایی که از `discovery` استفاده می‌کنند
افزودن/حذف را با خطای 409 رد می‌کنند، چون poller کشف‌سرویس منبع
حقیقت آن pool است و در tick بعدی خودش به‌صورت خاموش هر تغییر
اعمال‌شده توسط ادمین را بازنویسی می‌کرد.

**Drain/undrain** بدون توجه به نحوه پُرشدن بک‌اندهای یک pool روی
آن‌ها کار می‌کند، چون drain عضویت را تغییر نمی‌دهد — فقط
قابل‌انتخاب‌بودن را.

### معنای Draining

یک بک‌اند می‌تواند همزمان **سالم-و-در-حال-drain** باشد — این‌ها
محورهای مستقل‌اند. یک بک‌اند در حال drain:

- **هرگز انتخاب نمی‌شود** برای یک `Pick` جدید (نه توسط هیچ
  الگوریتمی، نه توسط تخصیص نشست sticky، و نه به‌عنوان fallback تازه
  وقتی یک کوکی sticky موجود نامعتبر می‌شود)
- اتصالات/درخواست‌های **در حال اجرا** را بدون دست‌زدن باقی می‌گذارد
  — اخراج فعال انجام نمی‌شود
- وقتی **همه** بک‌اندهای یک pool در حال drain باشند (یا ناسالم، یا
  هر دو)، `Pick` همان خطای موجود "هیچ بک‌اند سالمی موجود نیست" را
  برمی‌گرداند

### تغییرات فقط زمان اجرا (پایدار نشده)

افزودن‌ها و حذف‌های بک‌اند توسط ادمین **فقط زمان اجرا** هستند و
**هرگز** در فایل config نوشته نمی‌شوند. یک بازخوانی کامل بعدی
مبتنی بر فایل (SIGHUP یا polling فایل) عضویت pool را به هر چیزی که
فایل در حال حاضر می‌گوید بازنشانی می‌کند، و هر تغییر عضویت
اعمال‌شده توسط ادمین را کنار می‌گذارد. این به‌صورت عمدی است — فایل
config منبع حقیقت برای وضعیت اعلام‌شده است.

### داشبورد

داشبورد تعبیه‌شده از مسیر ریشه لیسنر ادمین (`/`) سرو می‌شود — مثلاً
`http://127.0.0.1:9091/`. آن را در مرورگر باز کنید:

- خود صفحه **بدون هیچ احراز هویتی** بار می‌شود — این فقط
  HTML/CSS/JS استاتیک است، که با `go:embed` در باینری تعبیه شده، با
  صفر وابستگی خارجی (بدون فونت یا آیکون یا کتابخانه JS از CDN؛
  به‌صورت کامل آفلاین کار می‌کند).
- در بار اول، صفحه توکن bearer ادمین را درخواست می‌کند (همان توکنی
  که از طریق `token_env` پیکربندی شده). توکن فقط در `sessionStorage`
  همان تب مرورگر ذخیره می‌شود — هرگز در `localStorage`، تا پس از
  راه‌اندازی مجدد مرورگر باقی نماند.
- هر تماس داده‌محوری که این صفحه برقرار می‌کند — هم تماس‌های REST
  پشت دکمه‌های drain/undrain و هم فید زنده زیر — به همان توکن نیاز
  دارد، دقیقاً مانند هر کلاینت دیگر API ادمین.
- این صفحه یک ردیف به‌ازای هر بک‌اند در همه pool‌ها نمایش می‌دهد:
  نام pool، آدرس، وزن، وضعیت سالم/در-حال-drain، و تعداد اتصال زنده.
  یک دکمه drain/undrain در هر ردیف مستقیماً همان نقاط پایانی REST
  موجود را صدا می‌زند — داشبورد یک فرانت‌اند سیم‌کشی‌شده به API
  ادمین است، نه یک صفحه کنترل جدید.

**به‌روزرسانی‌های زنده (WebSocket):** به‌جای اینکه مرورگر مرتباً
REST API را poll کند (که یعنی هر تب باز داشبورد به‌طور مستقل API
ادمین را زیر بار می‌برد)، داشبورد یک اتصال WebSocket واحد به
`/admin/ws` باز می‌کند و سرور هر ۱ تا ۲ ثانیه یک اسنپ‌شات JSON تازه
از وضعیت pool/backend را push می‌کند. این نقطه پایانی یک پیاده‌سازی
کاملاً دستی‌نوشته از پروتکل WebSocket (RFC 6455) است — handshake،
framing، همه‌چیز — چون این پروژه هیچ وابستگی خارجی نمی‌گیرد و
gorilla/websocket (یا مشابه آن) خارج از بحث است.

**نکته احراز هویت برای نقطه پایانی WebSocket:** API بومی WebSocket
مرورگرها هیچ راهی برای تنظیم یک هدر `Authorization` سفارشی روی
درخواست upgrade ندارد. به‌جای آن، توکن به‌عنوان یک پارامتر query
ارسال می‌شود — `/admin/ws?token=...` — و سمت سرور با همان مقایسه
با زمان ثابت که REST API استفاده می‌کند اعتبارسنجی می‌شود. چون توکن
در URL سفر می‌کند، می‌تواند در هرچیزی که URLهای خام درخواست را لاگ
می‌کند پایان یابد؛ لاگ‌نویسی خود Hoot-Lb هرگز این کار را نمی‌کند —
هر خط لاگ نزدیک به handler وب‌سوکت فقط مسیر درخواست را لاگ می‌کند،
هرگز رشته query را — اما اگر یک reverse proxy یا هر زیرساخت
لاگ‌نویسی دسترسی دیگری جلوی لیسنر ادمین قرار دهید، آن را تنظیم کنید
تا URLهای کامل را برای این نقطه پایانی لاگ نکند، وگرنه توکن را در
آن لاگ‌ها افشا خواهید کرد، حتی اگر خود Hoot-Lb این کار را نکند.

### محافظت هم‌زمانی

API ادمین یک سقف هم‌زمانی خود محافظتی دارد
(`max_concurrent_requests`، پیش‌فرض ۱۰). درخواست‌هایی که وقتی
سمافور پر است می‌رسند فوراً پاسخ 503 می‌گیرند — این یک صف عادلانه
نیست، بلکه یک سقف سخت برای جلوگیری از اتمام منابع است.

## ریستارت بدون داون‌تایم

برای تغییراتی که بارگذاری مجدد داغ نمی‌تواند مدیریت کند (آدرس/پروتکل/حالت
TLS لیسنر)، Hoot-Lb از ریستارت باینری بدون داون‌تایم از طریق الگوی
کلاسیک انتقال سوکت یونیکس پشتیبانی می‌کند: والد فایل‌های توصیفی (FD)
سوکت‌های گوش‌دهنده از قبل بایند شده را به فرزند تازه اجراشده منتقل
می‌کند، فرزند آن‌ها را مستقیماً به ارث می‌برد (بدون باز-بایند، بدون
race پورت-اشغال)، آمادگی را سیگنال می‌دهد، و فقط پس از آن والد
پذیرش را متوقف کرده و اتصالات در حال اجرای خود را تخلیه می‌کند.

این توسط موارد زیر فعال می‌شود:

- **SIGUSR2**: با `kill -USR2 <pid>` (همان قرارداد Unicorn/Puma).
- **Admin API**: `POST /admin/restart` (همان احراز هویت bearer-token
  هر نقطه پایانی دیگر ادمین). فوراً `202 Accepted` برمی‌گرداند؛
  انتقال به‌صورت ناهمگام است.

### ویژگی‌های ایمنی

- **Handshake آمادگی**: فرزند آمادگی را از طریق یک pipe *پیش از*
  بسته شدن هیچ‌یک از لیسنرهای والد سیگنال می‌دهد. اگر فرزند ظرف
  ۱۵ ثانیه شکست بخورد (config اشتباه، کرش)، والد انتقال را لغو
  کرده و دقیقاً همان‌طور که قبل بود به سرو کردن ادامه می‌دهد.
- **نگهبان تلاش منفرد**: یک SIGUSR2 یا تریگر ادمین دوم در حالی که
  یک انتقال در حال انجام است با یک خط لاگ واضح رد می‌شود.
- **ارتقای باینری**: چون `os.Executable()` استفاده می‌شود، ریستارت
  پس از قرار دادن یک باینری جدید روی دیسک، باینری *جدید* را برمی‌گرداند،
  نه قدیمی را.

### محدودیت پلتفرم

انتقال FD از طریق fork/exec فقط در لینوکس و داروین پشتیبانی
می‌شود. در پلتفرم‌های دیگر، تریگر ریستارت یک خطای واضح برمی‌گرداند.

### خارج از محدوده

وضعیت در حافظه (سطل‌های محدودکننده نرخ، شمارنده‌های قطع مدار،
شمارنده‌های متریک) در طول ریستارت **حفظ نمی‌شود** — فرزند با فقط
سوکت‌های گوش‌دهنده منتقل‌شده شروع تازه می‌کند. این مورد انتظار و
مستند شده است.

## تست (Testing)

این پروژه به‌طور انحصاری از چارچوب تست استاندارد Go استفاده می‌کند.
تمام تست‌ها به‌صورت پیش‌فرض با `-race` اجرا می‌شوند.

### اجرای تست‌ها

```sh
# مجموعه تست استاندارد (تمام بسته‌ها، با تشخیص نژاد)
go test -race ./...

# تست‌ها با تگ consul
go test -race -tags consul ./...

# بنچمارک‌ها (۳ ثانیه به‌ازای هر بنچمارک)
go test -bench=. -benchtime=3s ./internal/balancer/ ./internal/metrics/ \
    ./internal/proxy/l4/ ./internal/proxy/l7/

# تست بار (۱۰۰۰ درخواست همزمان، ادعای p99)
go test -tags load -run TestLoadTest100k -timeout 120s ./internal/proxy/l7/

# تست‌های آشوب (نیاز به ساخت باینری hoot-lb، ~۳۰ ثانیه هر کدام)
go test -tags chaos -timeout 300s ./internal/chaos/
```

### بنچمارک‌ها

مسیرهای حیاتی حساس به تأخیر بنچمارک می‌شوند:

| بنچمارک | بسته | چه چیزی را اندازه می‌گیرد |
|----------|------|--------------------------|
| `BenchmarkTCPRelay` | `proxy/l4` | توان رله TCP دوطرفه (پیلود 1KB–1MB) |
| `BenchmarkHTTPProxy` | `proxy/l7` | یک round-trip از طریق کل پشته L7 |
| `BenchmarkHTTPProxyParallel` | `proxy/l7` | پراکسی L7 تحت بار همزمان |
| `BenchmarkPickRoundRobin` | `balancer` | تأخیر Pick round-robin (۱۰ و ۱۰۰ بک‌اند) |
| `BenchmarkPickLeastConn` | `balancer` | تأخیر Pick least-connections (۱۰ و ۱۰۰ بک‌اند) |
| `BenchmarkCounterAdd` | `metrics` | افزایش اتمیک شمارنده (~1.6ns/op) |
| `BenchmarkHistogramObserve` | `metrics` | مشاهده اتمیک هیستوگرام (~2.9ns/op) |

### تست بار

`TestLoadTest100k` (تگ بیلد: `load`) ۱۰۰ goroutine همزمان راه‌اندازی
می‌کند، که هر کدام ۱۰۰۰ درخواست HTTP متوالی از طریق یک L7Server واقعی
با یک `httptest.Server` واقعی ارسال می‌کنند. ادعاها:

- صفر شکست در سراسر ۱۰۰,۰۰۰ درخواست.
- p99 latency کمتر از ۵۰ms (اندازه‌گیری شده از هیستوگرام مرتب‌شده).
- بدون نشت goroutine پس از اتمام تست.

### تست‌های آشوب

تست‌های آشوب (تگ بیلد: `chaos`) از `os/exec` واقعی برای شروع hoot-lb
به‌عنوان فرزند پردازه استفاده می‌کنند — هرگز mockهای درون‌پردازه‌ای.
هر سناریو تاب‌آوری تحت شرایط نامطلوب را آزمایش می‌کند:

| سناریو | چه چیزی را آزمایش می‌کند |
|--------|-------------------------|
| **شکست بک‌اند در وسط ترافیک** | کشتن یکی از ۳ بک‌اند در حین ترافیک پایدار؛ ادعای failover خودکار ظرف `health_check.interval * unhealthy_threshold` |
| **تکان دادن تنظیمات** | بازنویسی config + SIGHUP ۲۰ بار سریع در حین ترافیک؛ ادعای بالا بودن پردازه و نرخ خطا کمتر از ۱٪ |
| **ریستارت SIGUSR2 تحت بار** | تریگر ریستارت باینری بدون داون‌تایم با ۱۰ goroutine کلاینت همزمان به مدت ۱۰ ثانیه؛ ادعای صفر اتصال ریخته‌شده |
| **بازیابی اتمام منابع** | تنظیم `max_connections_per_listener: 50`، باز کردن ۲۰۰ اتصال، تأیید ۵۰ سرو شده / ۱۵۰ رد شده، سپس تأیید بازیابی پس از آزادسازی |

### تشخیص نشت goroutine

هر جزء که goroutine شروع می‌کند و متد `Stop()`/`Close()` دارد،
توسط تست نشت goroutine با الگوی خط پایه `runtime.NumGoroutine()`
پوشش داده شده است:

| جزء | تست نشت |
|------|---------|
| `l4.TCPServer` | `TestTCPProxyClientClose`, `TestTCPProxyManyConnections` |
| `l4.UDPServer` | `TestUDPProxyManyDatagrams` |
| `l4.TLSPassthroughServer` | `TestTLSPassthroughGoroutineLeak` |
| `l7.L7Server` | `TestL7ServerCloseGoroutineLeak` |
| `admin.Server` (WebSocket) | `TestDashboardWebSocketGoroutineLeak` |
| `health.Checker` | `TestHealthCheckerGoroutineLeak` |
| `reload.Watcher` | `TestWatcherStopGoroutineLeak` |
| `ratelimit.Limiter` | `TestStopGoroutineLeak` |
| `runtime.Poller` | `TestPollerStopGoroutineLeak` |
| `circuitbreaker.Breaker` | `TestCircuitBreakerChurn` |

## محدودیت‌های شناخته‌شده

- **کشف سرویس etcd وجود ندارد.** فقط DNS و Consul (از طریق تگ بیلد)
  پشتیبانی می‌شوند. ادغام etcd ممکن است در نسخه آینده اضافه شود.
- **وضعیت در حافظه در طول ریستارت حفظ نمی‌شود.** سطل‌های محدودکننده
  نرخ، شمارنده‌های قطع مدار، و شمارنده‌های متریک هنگام ریستارت پردازه
  (حتی در حین انتقال FD بدون داون‌تایم) بازنشانی می‌شوند.
- **کشف DNS نمی‌تواند TTL را به‌صورت خودکار تشخیص دهد.** اپراتور
  باید `refresh_interval` را برابر یا کمتر از TTL واقعی DNS خود
  تنظیم کند.
- **تخلیه فعال اتصالات هنگام حذف بک‌اند وجود ندارد.** وقتی یک
  بک‌اند حذف می‌شود، اتصالات در حال اجرای موجود به‌صورت طبیعی
  اجازه تکمیل دارند — تخلیه اجباری اتصالات فعال انجام نمی‌شود.

## نقشه راه (Roadmap)

کارهای آینده ممکن است شامل موارد زیر باشد:

- آداپتورهای کشف سرویس اضافی (etcd، Consul long-poll)
- تخلیه اتصالات با اخراج قابل‌تنظیم هنگام حذف بک‌اند
- صف‌بندی درخواست‌ها وقتی همه بک‌اندها در ظرفیت کامل هستند

## متریک‌ها

Hoot-Lb متریک‌های سازگار با Prometheus را روی یک لیسنر HTTP اختصاصی،
جدا از ترافیک پراکسی‌شده، عرضه می‌کند. به‌صورت پیش‌فرض فقط روی
loopback بایند می‌شود (`127.0.0.1:9090`).

### تنظیمات

```yaml
global:
  metrics:
    enabled: true            # پیش‌فرض true
    address: "127.0.0.1:9090"
    path: "/metrics"          # پیش‌فرض "/metrics"
```

### متریک‌های عرضه‌شده

| متریک | نوع | برچسب‌ها | توضیح |
|--------|------|--------|-------------|
| `lb_connections_total` | Counter | listener, pool, backend, protocol | تعداد کل اتصالات برقرارشده |
| `lb_connections_active` | Gauge | listener, pool, backend, protocol | اتصالات فعلاً فعال |
| `lb_request_duration_seconds` | Histogram | listener, pool, backend | مدت درخواست لایه ۷ (سطل‌های ثابت: 5ms–10s) |
| `lb_bytes_transferred_total` | Counter | listener, pool, backend, direction | بایت‌های منتقل‌شده (جهت: upstream/downstream) |
| `lb_dial_failures_total` | Counter | listener, pool, backend | تعداد کل شکست‌های dial |
| `lb_backend_healthy` | Gauge | pool, backend | وضعیت سلامت بک‌اند (1=سالم, 0=ناسالم) |

### کارایی مسیر داغ (Hot path)

همه عملیات افزایش Counter/Gauge/Histogram فقط از عملیات اتمیک
استفاده می‌کنند — بدون mutex روی مسیر داغ. بنچمارک‌ها روی Apple M4
حدود ۱.۶ نانوثانیه به‌ازای هر عملیات برای Counter/Gauge و ۲.۹
نانوثانیه برای Histogram نشان می‌دهند.

### پاکسازی Cardinality

وقتی بک‌اندها از یک pool حذف می‌شوند (از طریق به‌روزرسانی discovery
یا بازخوانی config)، ورودی‌های برچسب متریک آن‌ها به‌صورت خودکار
پاکسازی می‌شود تا از رشد بی‌حد cardinality جلوگیری شود.

## لاگ دسترسی (Access logging)

لاگ‌های دسترسی JSON ساخت‌یافته برای هر اتصال کامل‌شده (لایه ۴) یا
درخواست (لایه ۷) به stdout منتشر می‌شوند.

### تنظیمات

```yaml
global:
  access_log:
    enabled: true
    format: json              # فعلاً فقط "json"
```

### فیلدها

| فیلد | نوع | توضیح |
|-------|------|-------------|
| `timestamp` | string | برچسب زمانی RFC3339Nano |
| `listener` | string | نام لیسنر |
| `pool` | string | نام pool بک‌اند |
| `backend` | string | آدرس بک‌اند |
| `protocol` | string | پروتکل (tcp، udp، http) |
| `client_addr` | string | آدرس IP کلاینت |
| `duration_ms` | int | مدت اتصال/درخواست به میلی‌ثانیه |
| `bytes_sent` | int | بایت‌های ارسال‌شده به upstream |
| `bytes_received` | int | بایت‌های دریافت‌شده از downstream |
| `method` | string | متد HTTP (فقط لایه ۷، برای لایه ۴ حذف می‌شود) |
| `path` | string | مسیر HTTP (فقط لایه ۷، برای لایه ۴ حذف می‌شود) |
| `status_code` | int | کد وضعیت HTTP (فقط لایه ۷، برای لایه ۴ حذف می‌شود) |
